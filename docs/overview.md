# EVER-STAR 프로젝트 개요

## 서비스 한줄 소개
반려동물과의 추억을 기록‧추모하는 **EVER-STAR** 웹 서비스. 지구별(메인)에서 편지 작성/조회, 영원별(추모별)에서 추모 활동 및 메모리얼북을 관리합니다.

## 핵심 기능
- **지구별 메인**: 편지 쓰기/편지함, 마이페이지 진입.
- **영원별(Everstar)**: 추모별 메인 뷰, 응원 메시지, 탐험/검색.
- **메모리얼북**: 활성/비활성 상태 확인 및 내용 조회.
- **반응형 UI**: 모바일/태블릿/데스크탑 대응.
- **E2E 스크린샷**: 주요 화면(00~14번) 자동 캡처.

## 기술 스택(프론트엔드)
- React, TypeScript
- Playwright(E2E)
- Tailwind 기반 스타일(구성요소는 레이아웃 코드 참고)
- MSW(Mock Service Worker)로 로컬/테스트 데이터 모킹

## 개발/테스트 가이드
```bash
cd everStarFrontend
npm install
npm run dev           # http://localhost:3000
npm run test:e2e      # Playwright 전체 E2E 및 스크린샷 생성
```

# 김도이

EVERSTAR는 반려동물이 무지개다리를 건넌 후, 펫로스 증후군으로 힘들어하는 사람을 돕기 위한 서비스입니다. 이번 프로젝트는 저에게 있어 많은 의미가 있는 프로젝트 입니다. 개발을 배운 후 처음 참여하는 프로젝트 이기 때문입니다. 또한 spring을 처음 써보는 경험이기도 하여 제게 큰 도전 이었습니다. 기획부터 배포까지 처음 경험 하여 부족함이 많았습니다. 그럼에도 훌륭한 팀원들 덕분에 가시적인 성과를 얻어 영광입니다.

**느낀점**

이번 프로젝트를 통해 팀워크의 중요성을 다시 한번 느꼈으며, 뛰어난 팀원들과 함께 협업 하면서 많은 것을 배울 수 있는 감사한 기회였습니다.

**아쉬운 점**

- 프로젝트 기간이 짧아서 다른 모든 팀원들이 구현한 코드나 기술들을 모두 이해하며 진행하지 못했습니다. 제가 맡은 부분에 대한 공부 하고 구현하기에도 급했기 때문입니다.
- 본격적인 프로젝트를 처음 하다보니 git 활용중 어려움이 생기는 경우가 있었습다. 충돌 등 git 문제를 해결하느라 날린 시간이 많았습니다. 따라서 추후에는 git 활용법에 대해 더 공부하고 사용 해야겠습니다.

**잘한점**

- BDD 테스트 수행
CI/CD 단계에서 단위테스트를 통한 자동 배포를 했습니다. 테스트를 작성하는것이 익숙치 않고, 생각보다 시간이 많이 들어 힘들었습니다. 하지만 테스트 덕분에 프론트에서 API 연결시 에러가 거의 발생하지 않았습니다. 이를 통해 코드의 안정성에서나, 협업 관점에서나 단위 테스트의 중요성을 몸소 알게 되었습니다.
- 코드리뷰
다른 팀들을 살펴보니 코드리뷰를 통한 Merge를 하는 팀이 없었습니다. 저희팀은 유일하게 코드리뷰를 동반한 Merge Request를 하여 코치님들께 칭찬을 받았습니다. 이 덕분에 휴먼에러를 미리 방지할 수 있었고 팀원들이 어떤 내용을 구현하는지 파악하거나 컨벤션을 유지하기 용이했습니다.
코드리뷰는 잊어버리기 전에 바로바로 반영 했습니다. 미뤄질 경우 작성자와 리뷰자가 리뷰내용과 이유를 잊어버릴 수 있기 때문입니다.

## 1. 결합도 분리

코드 사이에 강한 결합이 있으면, 한 부분이 변경되었을 때 다른 부분에도 영향을 많이 주게 됩니다.  반대로 낮은 결합도를 유지하게 된다면 유지보수가 용이할 뿐만 아니라 재사용성이 증가하고, 테스트 또한 편리해집니다. 그래서 개발을 진행하며 객체간의 결합도를 낮추기 위해 노력했습니다.

![image.png](images/image_0048.png)

스프링에서는 의존성 주입으로 결합도를 낮출 수 있습니다. 엔티티 객체와 DTO, 컨트롤러, 서비스, 리포지토리를 기본적으로 나누고 의존성 주입을 적극적으로 활용하였습니다. 또한 OpenAI 요청과 같은 소요 시간이 오래 걸리는 로직은 SpringBoot의 EventListener를 활용하여 별도로 처리하도록 하여 Client에 응답이 빠르게 갈 수 있도록 하였습니다.

## 2. JPA, QueryDSL 활용

JPA는 Java 객체를 관계형 데이터베이스에 매핑하는 ORM(Object-Relational Mapping)기술 표준으로 사용되는 인터페이스의 모음입니다. 즉, 실제적으로 구현된 것이 아니라 구현된 클래스와 매핑을 해주기 위해 사용되는 프레임워크입니다.

 저희는 SQL문이 아닌 Method를 통해 DB를 조작할 수 있는 JPA를 활용하여 비즈니스 로직을 구성하는 데에 집중하여 객체지향적인 코드를 개발하려고 하였습니다. 또한, JPA 메소드만으로 해결하기 힘든 복잡한 동적쿼리가 필요한 경우에는 QueryDSL을 활용하여 쿼리문을 작성하여 DB에 요청을 보냈습니다.

또한 JPA로 구성된 비즈니스 로직을 개발 후, 로그를 확인하여 N+ 1문제가 발생하는 경우가 있는지 확인하고, 있다면 `@EntityGraph` 를 활용하여 이 문제를 해결하였습니다.

## 3. Scheduler 활용

SpringBoot에서 제공하는 스케쥴러 기능은 아래와 같이 크게 두 가지가 있습니다.

**`@Scheduled`**: 간단한 주기적 작업 스케줄링을 위해 사용. 설정이 쉽고 빠르게 설정 가능함

**`TaskScheduler`**: 복잡한 스케줄링이나 동적 스케줄 변경이 필요한 경우 사용. 유연성과 확장성이 높지만 설정이 상대적으로 복잡함

EVERSTAR 서비스에서는 사용자가 퀘스트를 완료하고 다음날 특정 시간에만 로직이 수행되도록 해야했습니다. 그래서 주기적으로 스케쥴링이 되는 것보다는 사용자가 지정한 시간을 기준으로 한 번만 스케쥴링이 되어 로직이 수행되어야하기 때문에 TaskScheduler를 사용했습니다.

## 4. 단위테스트 수행

단위테스트는 코드의 가장 작은 기능적 단위를 테스트하는 프로세스입니다.

저희는 안정적인 서비스 개발을 하기 위해 기능을 개발할 때마다 단위테스트 코드도 작성하였습니다. 성공, 실패의 경우에 대한 테스트를 작성하면서 개발한 로직이 정상적으로 작동하는지 확인하였습니다.

테스트를 작성한 덕분에 개발 도중 기능이 변경되거나, 리팩토링을 하게 되는 경우에 바뀐 로직때문에 동작이 되지 않는 부분을 빠르게 확인하고 수정할 수 있었습니다. 또한 Jenkins를 기반으로 한 CI/CD 환경을 구축하고 개발을 하였는데, 테스트 코드를 통해 안정적인 CI가 이루어질 수 있었습니다.

## 트러블 슈팅

- 강세현
    - **문제 상황1**
    
    ```java
    @Service
    @RequiredArgsConstructor
    @Transactional(readOnly = true)
    @Slf4j
    public class QuestAnswerService {
    
    	private final TaskScheduler taskScheduler;
    	private final QuestAnswerRepository questAnswerRepository;
    	private final PetRepository petRepository;
    	private final MemorialBookRepository memorialBookRepository;
    	private final MemorialBookService memorialBookService;
    
    	@Transactional
    	public void createQuestAnswer(Authentication authentication, Long petId) {
    		User user = ((PrincipalDetails) authentication.getPrincipal()).getUser();
    
    		// TODO: 질문 답변 생성 메서드 구현
    
    		plusPetQuestIndex(user, petId);
    	}
    
    	private void plusPetQuestIndex(User user, Long petId) {
    		Optional<Pet> findPet = petRepository.findById(petId);
    		Pet pet = findPet
    			.orElseThrow(() -> new ExceptionResponse(CustomException.NOT_FOUND_PET_EXCEPTION));
    
    		pet.plusQuestIndex();
    
    		if (pet.getQuestIndex() == 49) {
    			scheduleMemorialBookActivation(user, pet);
    		}
    	}
    
    	private void scheduleMemorialBookActivation(User user, Pet pet) {
    		LocalTime questReceptionTime = user.getQuestReceptionTime();
    		LocalDateTime nextDayQuestReceptionTime = LocalDateTime.of(LocalDate.now().plusDays(1), questReceptionTime);
    
    		Date nextDayQuestReceptionDate = Date.from(nextDayQuestReceptionTime.atZone(ZoneId.systemDefault()).toInstant());
    		log.info("nextDayQuestReceptionDate : {}", nextDayQuestReceptionDate);
    		taskScheduler.schedule(() -> memorialBookService.changeActiveStatus(pet.getId()), nextDayQuestReceptionDate);
    	}
    }
    ```
    
    49번째 질문-답변이 끝나고 일정 시간 뒤에 Spring Scheduler를 사용하여 메모리얼북의 활성화 상태를 변경해야 함.
    
    그런데 Entity인 `MemorialBook`을 `MemorialBookRepository`에서 받아오고 `MemorialBook`의 
    `changeActiveStatus` 를 호출하는 경우, 트랜잭션이 적용되지 않아 DB에 변경사항이 저장되지 않음.
    
    그래서 `MemorialBookService` 내의 트랜잭션이 적용되는 `changeActiveStatus` 메소드를 호출하여 DB에 변경사항을 적용하려고 하는데, 이 경우 Service가 Service를 호출하여 추후 순환참조를 하게 되는 경우가 발생할 수 있음.

    Spring Scheduler를 사용하여 MemorialBook 관련 스케줄링 하는 코드를 별도의 스케줄러 클래스로 분리하고 `@Component` 를 사용하여 컴포넌트로 만들고 이를 사용해서 스케줄링하도록 함.
    
    ```java
    
    @Component
    @RequiredArgsConstructor
    public class MemorialBookScheduler {
    
    	private final TaskScheduler taskScheduler;
    	private final MemorialBookService memorialBookService;
    
    	public void scheduleMemorialBookActivation(User user, Long petId) {
    		LocalTime questReceptionTime = user.getQuestReceptionTime();
    		LocalDateTime nextDayQuestReceptionTime = LocalDateTime.of(LocalDate.now().plusDays(1), questReceptionTime);
    		Date nextDayQuestReceptionDate = Date.from(nextDayQuestReceptionTime.atZone(ZoneId.systemDefault()).toInstant());
    
    		taskScheduler.schedule(() -> memorialBookService.changeActiveStatus(petId), nextDayQuestReceptionDate);
    	}
    }
    ```

    SpringBoot 기본 설정의 TaskScheduler 스레드풀 사이즈는 1로 고정되어 있음
    
    즉, 하나의 스레드에서 여러 작업이 대기함
    
    하나의 스레드에서 여러 작업이 대기하면 들어온 순서대로 실행되며, 하나의 스레드에서 모든 작업을 수행하면 처리 속도가 늦어짐.
    
    예를 들어 여러 작업이 동일한 시간에 예약되었을 때, 하나의 스레드에서 순차적으로 작업을 수행하면 처리 속도가 늦어질 수 있음.

# 정현조

### 1. 프로젝트 개요

에버스타는 반려동물의 상실(펫로스) 경험을 돕기 위한 서비스입니다. 이번 프로젝트는 저에게 있어 많은 도전과 성장이 담긴 경험이었습니다. 저는 프론트엔드 개발자이자 팀장으로서 프로젝트를 이끌며, 뛰어난 팀원들과 협력하는 과정에서 다양한 어려움을 극복해나갔습니다.

### 2. 기획 및 설계 과정

### 2.1 기획 단계

프로젝트 초기 기획 단계는 순탄하지 않았습니다. 수많은 아이디어를 제안했지만, 그 중 다수가 반려되었고, 프로젝트의 방향성을 잡기까지 시간이 필요했습니다. 특히 1주차가 지나갈 때쯤에서야 비로소 구체적인 목표와 방향을 설정할 수 있었습니다

### 2.2 설계 단계

설계 과정에서 더 큰 도전에 직면했습니다. 서비스 플로우를 구상하고 필요한 페이지와 기능을 정하면서 방대한 확장성을 고려했지만, 세부적인 부분에서 팀원 간의 합의가 부족해 의견 충돌이 발생했습니다. 특히, 피그마(Figma)에서 귀여운 디자인을 시도했으나, 초기 결과물이 만족스럽지 않아 이틀 밤을 새우며 수정 작업을 해야 했습니다. 디자인에 지나치게 신경 쓰다 보니 기능 구현 부분의 설계가 부족해지는 문제도 발생했습니다.

### 3. 개발 과정

### 3.1 기술적 도전

개발 단계에서는 처음 접하는 TypeScript(ts), Storybook, 그리고 React를 활용한 개발이 정말 커다란 도전이었습니다. 우리는 아토믹 디자인 구조를 채택했지만,  앞서 말한 디자인 단계가 길어지면서 구조를 완벽하게 나누지 못한 채 작업을 시작했습니다. 이로 인해 팀원들이 불편함을 느끼기도 했지만, 이러한 과정을 통해 아토믹 디자인에 대한 이해를 깊게 할 수 있었습니다. 또한 Storybook을 활용한 시각적 테스트는 React와 TypeScript를 학습하는 데 큰 도움이 되었고, 개인적으로도 많은 성장을 이룰 수 있었습니다.

### 3.2 팀원들의 기여

다행히도 프론트엔드의 복잡한 기술적 부분, 예를 들어 웹소켓, 오픈비두(OpenVidu), FCM, SSE 등은 뛰어난 팀원들이 맡아주었습니다. 저는 비교적 간단한 OAuth 연동이나 UI/UX 개선 작업에 집중할 수 있었습니다. 이를 통해 팀원들의 협력과 분업이 얼마나 중요한지를 깨달았으며, 각자의 강점을 살려 프로젝트를 성공적으로 마무리할 수 있었습니다.

### 4. 프로젝트 마무리 및 성과

프로젝트의 마지막 단계에서, 저는 개인 사정으로 인해 수상 후 발표회 준비 과정에 참석하지 못했지만, 팀원들이 훌륭하게 발표를 진행해주었고, 그 결과 우수한 성과를 거둘 수 있었습니다. 이번 프로젝트를 통해 팀워크의 중요성을 다시 한번 느꼈으며, 뛰어난 팀원들과 함께 일하면서 많은 것을 배울 수 있었습니다.

## 1. PWA 적용

PWA(Progressive Web Apps)는 **모바일 기기에서 네이티브 앱과 같은 사용자 경험을 제공하는 웹 앱**을 의미합니다. EVERSTAR는 사용자에게 특정시간에 알림을 주는 기능이 있는데 데스크탑보다는 모바일이 사용자에게 푸시알림을 주기에 나았습니다. 퀘스트를 위해 사진을 첨부할 때 주소를 치고 웹사이트에 들어가서 이를 업로드하는 번거로운 방식보다 모바일 앱처럼 바로 사이트에 접근하여 퀘스트를 하는 것이 사용자 편의성 측면에서 유리하다는 판단으로 PWA를 적용하게 되었습니다. 

https로 배포되어 있었기 때문에 바로  `Web App Manifest 파일` (PWA가 사용자의 디바이스에 어떻게 표시되고 동작해야 하는지 브라우저에게 알려주는 JSON 파일)을 작성하고 `192x192` 와 `512x512` 크기의 아이콘을 넣어서 앱에서 보여질 아이콘을 만들 수 있었습니다. 

그리고 `service-worker.js`(애플리케이션의 백그라운드에서 실행되는 스크립트. Service Worker는 리액트 애플리케이션을 오프라인에서 사용할 수 있게 한다), `serviceWorkerRegistration.js` 파일(서비스 워커가 성공적으로 등록되었는지 여부를 알려준다)을 작성하여 해당 리액트 애플리케이션이 PWA 로 기능할 수 있게 하였습니다.

직접 WebRTC를 구현하기 보다 사용이 용이한 Openvidu 라이브러리를 적용해서 화상 채팅을 구현하기로 했습니다. 우리가 원하는 비디오, 마이크, 스피커 on/off 기능과 초대 기능을 모두 손쉽게 구현할 수 있었습니다.

![openvidu.png](images/image_0048.png)

 Chatting은 웹소켓을 이용해서 구현하였습니다. openvidu에서 세션이 생성되면 채팅 세션을 열어서 서로 메시지를 주고받을 수 있게 하였습니다.

## 3. FCM

FCM을 이용해 사용자에게 반려동물로부터 편지 답장이 왔음을 알리는 기능을 구현했습니다. 

이 기술을 사용함으로써, iOS, Android, 웹 등 다양한 플랫폼에 관계없이 동일한 방식으로 푸쉬 메시지를 전송할 수 있었습니다. 또한 webSocket으로 푸쉬메시지를 실시간으로 수신하려면 서버와 지속적인 연결로인해 네트워크 부하가 생기지만, FCM 서버를 중간에 둠으로써 이 문제를 해결하고, 클라이언트는 FCM 서버로부터 필요할 때에만 메시지 수신이 가능하도록 하였습니다.

## 4. SSE

**SSE(Server Sent Event)** 란 시간으로 서버에서 클라이언트로 데이터를 보내는 단방향 통신을 의미합니다. 로그인 후 EarthMain 페이지에서 특정 시간에 사용자에게 퀘스트 알림을 보내고, 퀘스트 완료시에 상태를 반영할 필요가 있었습니다. 

이에 SSE를 이용해 서버로부터 알림을 수신할 수 있게 구현하였습니다. 

## 5. 아토믹 디자인

![image.png](images/image_0048.png)

컴포넌트를 만들 때 중복 작성하는 문제를 해결하기 위해 아토믹 디자인을 적용했습니다. 컴포넌트를 atoms, molecules, organisms, templates, pages의 5단계로 나눴고, atoms을 합쳐 molecules를 만드는 등 단계별로 컴포넌트를 결합해서 전체 페이지를 구현했습니다.

이 과정에서 가장 큰 문제는 각 단계의 차이를 정하는 것이었는데, 팀원 간의 대화를 통해 아래와 같이 합의된 기준을 세워서 이를 극복하였습니다.

```jsx
1. Atoms: 더이상 나눌 수 없는 최소 단위 ex) button, input, label 등
2. Molecules: 컨텍스트 없이 UI 적인 요소로 SRP를 지킬 수 있는 컴포넌트 ex) LetterCard, ImageBox
3. Organisms: 작성한 컴포넌트에 컨텍스트가 있는 경우 ex) CheerMessage, PetInfoTab
4. Templates: Api 연결 이전의 와이어 프레임
5. Pages: Api 연결한 최종 페이지
```

또한 화면 이동, API 연결은 후순위로 미루고 일단 컴포넌트 뷰를 만들었는데, 나중에 연결작업을 할 때 어느 단계에서 연결을 할지가 문제가 되었습니다. 이에 Organisms 단계부터 주로 비즈니스 로직을 두는 식으로 합의하였습니다.

이렇게 컴포넌트를 만드는 과정에서 스토리북을 적극적으로 활용했습니다. 이에 스토리북에서 컴포넌트의 상태와 동작을 한눈에 확인할 수 있고 유연성과 재사용성이 높아지는 장점이 있습니다.

```java
<configuration>
    <!--  <appender> KafkaAppender 클래스를 사용하여 Kafka에 로그 메시지를 전송  -->
    <appender name="LOG-KAFKA" class="com.github.danielwegener.logback.kafka.KafkaAppender">
        <!--  <encoder> 로그 메시지를 Kafka가 처리할 수 있는 형식으로 변환  -->
        <encoder class="com.github.danielwegener.logback.kafka.encoding.LayoutKafkaMessageEncoder">
            <!--  <layout> 로그 메시지 출력 패턴 지정  -->
            <layout class="ch.qos.logback.classic.PatternLayout">
                <pattern>%date - %-5p %t %-25logger{5} %F:%L %m%n</pattern>
            </layout>
        </encoder>
        <!-- <topic> Kafka에 전송될 토픽의 이름을 지정  -->
        <topic>ek-log</topic>
        <!-- <keyingStrategy> 키 생성 전략을 Round Robin으로 설정 (모든 파티션에 균등하게 분산되도록 메시지들이 순환적으로 파티션에 할당)  -->
        <keyingStrategy class="com.github.danielwegener.logback.kafka.keying.RoundRobinKeyingStrategy"/>
        <!-- <deliveryStrategy> 비동기적 방식으로 로그 메시지 전송  -->
        <deliveryStrategy class="com.github.danielwegener.logback.kafka.delivery.AsynchronousDeliveryStrategy"/>
        <!-- <producerConfig> Kafka producer의 설정값 (순서대로, 재전송 횟수, 전송 서버 주소, 압축 타입, 데이터 전송 최대 대기 시간  -->
        <producerConfig>retries=1</producerConfig>
        <producerConfig>bootstrap.servers= </producerConfig>
        <producerConfig>compression.type=snappy</producerConfig>
        <producerConfig>max.block.ms=1000</producerConfig>
    </appender>

    <!-- <appender> 콘솔에 로그를 출력하기 위한 appender  -->
    <appender name="STDOUT" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>%date - %-5p %t %-25logger{5} %F:%L %m%n</pattern>
        </encoder>
    </appender>

    <!--
    <logger> elk 이름의 logger를 정의하여, INFO 레벨 메시지만 처리
    <appender-ref> elk 이름의 logger가 사용할 appender 지정
     -->
    <logger name="elk" level="INFO" additivity="false">
        <appender-ref ref="LOG-KAFKA"/>
        <appender-ref ref="STDOUT"/>
    </logger>

    <!-- <root> 모든 logger에 대한 기본 설정을 정의 (INFO 레벨 이상의 메시지만 처리) -->
    <root level="INFO">
        <appender-ref ref="STDOUT"/>
    </root>
</configuration>
```
