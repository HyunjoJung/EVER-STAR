# 이학범 수의사 인터뷰 기사

이학범 수의사가 말하는 펫로스 증후군 대처법

**Q. 펫로스 증후군을 진단하는 기준은 무엇인가.**

A. “반려동물과 이별을 겪으면 슬픈 게 당연하다. 다만 일정 시간이 지나도 감정이 회복되지 않고 일상으로 복귀하기 어렵다면 펫로스 증후군일 개연성이 크다. 펫로스 증후군에 비견되는 정신과 질환으로는 ‘지속성 복합 사별장애’가 있다. 가족, 친구 등 소중한 사람을 떠나보내고 꽤 오랜 시간(성인 기준 1년 이상)이 지났음에도 강도 높은 우울감이 지속되면 이 질환을 의심한다. 펫로스 증후군에도 같은 기준을 적용할 수 있다. 자가진단 체크리스트를 해보는 것도 방법이다(표 참조).”

**Q. 어떤 반려인에게 펫로스 증후군이 나타나기 쉬운가.**

A. “반려동물과 애착 관계가 깊을수록 떠나보내는 과정이 힘들다. 우울증 등 정신질환을 앓거나 사회에서 소외당한 경험이 있는 분들 가운데 반려동물을 입양해 키우며 의지하는 경우가 있다. 이런 반려인은 반려동물을 ‘유일한 내 편’으로 인식해 더 특별한 관계를 맺는다. 반려동물의 의미가 남달랐던 만큼 이별 과정에서 기존 질환이 악화되거나 마음을 크게 다칠 수 있어 특히 조심해야 한다.”

**Q. 펫로스 증후군을 방치하면 어떤 문제가 발생하나.**

A. “일반적으로 상실은 부정→분노→타협→절망→수용 다섯 단계로 진행된다. 펫로스 증후군은 반려동물과 이별을 납득하는 수용 단계로 나아가지 못하고 부정, 분노, 타협, 절망이 오랫동안 지속되는 것을 말한다. 분노 단계에선 가정불화가 빈번하게 발생한다. ‘내가 사람 먹는 음식 주지 말랬잖아’ 이런 식으로 가족에게 분노의 화살을 돌리기 때문이다. 더 위험한 건 절망 단계다. 반려동물의 부재에 망연자실해 극단적 선택을 한 사례가 실제 있다. 결코 가볍게 볼 문제가 아니다.”

**Q. 이별의 슬픔을 건강하게 극복할 수 있는 방법을 소개한다면.**

A. “일단 슬픈 게 정상이라고 얘기하고 싶다. ‘내가 너무 유난스러운 걸까’ ‘감정을 드러냈다가 핀잔을 듣진 않을까’ 생각하지 말고 이별 직후 자신의 속도대로 충분히 슬퍼했으면 한다. 감정을 억누르면 안에서 곪기 때문이다. 구체적으로는 떠난 반려동물에게 편지를 쓰면 좋다. 글로 마음을 전하는 과정에서 감정을 잘 정리할 수 있다. 반려동물 생전에 찍어놓은 사진, 영상을 본다든가 가족 혹은 반려동물을 키우는 주변 사람에게 힘든 마음을 털어놓는 것도 도움이 된다. 여러 방법을 써봤는데도 우울감이 오래 지속된다면 정신과 전문의에게 상담받는 걸 권한다.”

**Q. 더는 반려동물을 키우지 않으려 하거나 떠난 반려동물과 닮은 반려동물을 입양하는 사람도 많은데.**

A. “펫로스 증후군에는 죄책감, 후회 감정이 크게 작용한다. ‘우리 코코가 어디 가는 걸 참 좋아했는데’ ‘무슨 간식을 잘 먹었는데’ ‘마지막에 외출도 못 하고 처방사료만 먹다 갔네’ 이런 식이다. 후회를 줄이기 위해 반려동물 컨디션에 무리가 안 가는 선에서 평소 좋아하던 걸 다 하게 해주길 추천한다. 또 하나는 반려동물의 장례를 어떻게 치를지 미리 생각해두는 것이다. 경황이 없어 사체를 일반폐기물로 처리하고 나면 두고두고 마음에 짐처럼 남을 수 있다. 합법적이면서 정서적으로 납득 가능한 사체 처리 방법인 장례를 준비하는 것이 좋다.”

A. “둘 다 이별 고통이 커서 그런 건데, 전자의 경우 반려동물이 주는 행복이 얼마나 큰지 아는 사람으로서 안타깝다. 추천하는 방법은 첫째 반려동물이 노령견(노령묘)이 됐을 때 유기동물보호소 등에서 둘째 반려동물을 입양해 같이 지내는 것이다. 그러면 둘째를 자연스럽게 받아들이는 것은 물론, 서로 위로하면서 슬픔도 더 쉽게 극복할 수 있다. 같은 견종(묘종)의 닮은 반려동물을 입양하는 건 개인적으로 말리고 싶다. 대체품 찾듯 입양하면 첫째 반려동물과 다른 점이 보일 때 쉽게 실망하고 그런 마음이 파양으로까지 이어질 수 있기 때문이다. ‘해피는 안 그랬는데 얘는 왜 이러지’ 싶은 생각이 든다는 얘기다.”

**Q.반려동물을 여러 마리 키울 경우 남겨진 다른 반려동물의 슬픔에는 어떻게 대처해야 하나.**

A. “슬픔을 느끼는 정도는 반려동물마다 다르다. 평소와 완전히 똑같은 반려동물도 있다. 반대로 첫째 반려동물이 떠난 후 식욕이 없어진다거나, 밤에 잠을 잘 못 잔다거나, 첫째가 머무르던 자리를 떠나지 못한다거나 하는 증상을 보이면 마음이 힘든 상태라고 보면 된다. 심할 경우 공격성, 분리불안을 보이는 반려동물도 있다. 이때는 반려인이 함께 시간을 많이 보내면서 첫째 반려동물의 빈자리를 채워주는 게 가장 좋다. 분리불안 같은 증상은 수의사의 진료를 받아보는 것도 도움이 된다.”

**Q. 펫로스 증후군을 겪는 사람이 도움을 받을 만한 곳이 있다면.**

A. “아직까지 정부나 지방자치단체 차원의 지원 제도는 없다. 다만 최근 들어 펫로스 증후군을 전문으로 상담하는 정신과 전문의와 심리상담사가 늘고 있다. 펫로스 증후군을 겪는 반려인 모임도 민간에서 활발하게 운영되고 있다. 반려동물 장례식장에서 이런 모임을 연결해주기도 하니 문의하면 된다.”

**Q. 수의사도 펫로스 증후군을 겪나. 이별을 준비하는 반려인에게 당부하고 싶은 말이 있다면.**

# 펫로스_민경_인터뷰

**Q. 간단한 자기소개 부탁드립니다.**

**Q. 옛날에 키우던 반려동물을 소개해주실 수 있나요?**

A. 제 반려동물은 고슴도치였구요. 이름은 도치고 남자 아이였습니다.

**Q. 지금 반려동물과 함께 살고 계신가요?**

A. 지금은 자취를 하고 있어서 같이 살지는 못하는데 본가에 가면 강아지가 있습니다.

**Q. 제가 알기로는 반려동물 도치가 무지개 다리를 건넌 것으로 알고 있는데 그때의 상황을 설명해주실 수 있으신가요?**

A. 제가 초등학교 때 도치를 키웠는데 도치가 엄청 작았어요. 다른 애들에 비해서 애가 엄청 작아서 귀엽고 잘 지냈는데 어느순간 애가 아프더라구요. 그래서 아빠가 병원도 자주 데려가고 그랬는데, 그 날 딱 가족끼리 외식을 하러 갔다가 집에 돌아왔는데 도치가 혼자 경련을 일으키면서 뻗어있는거에요. 그래서 깜짝 놀라서 다 같이 동물병원에 간 기억이 있어요.

**Q. 조금 조심스럽지만 반려동물과 이별을 겪었을 때 감정을 설명해주실 수 있으신가요?**

A. 감정은 좀 어려서 구체적으로 기억이 나진 않는데 제가 계속 울었던 건 기억이 나요. 제가 어릴 때부터 강아지를 키우고 싶다는 생각이 많았는데 부모님이 강아지는 어려울 것 같다고 하셔서 고슴도치를 분양받아왔던 건데, 강아지처럼 제가 매일 손에 데리고 다니고 좋아해서 **엄청 유대감이 있었던 것 같아요. 그래서 계속 울고 3일 정도는 밥도 안 먹고 그 초등학생 아기가, 그랬던 걸 부모님께 들었었습니다.**

**Q. 그러면 보통 많은 반려인분들이 반려동물과 이별한 후, “있을 때 잘해줄 걸”와 같이 후회한다는 말을 많이 하세요. 그래서 민경씨도 도치가 떠난 후에 후회하셨던 경험이 있으신가요?**

A. 음 약간 걔가 한 번씩 가시를 뽀족하게 세울 때가 있어요. 그러면 괜히 저는 아파서 도치한테 화내고 짜증내고... 그리고 이제 그 때는 어렸을 때니까 같이 있다가 다른 친구들 만나러 가거나... 되게 무관심해질 그런 시기가 있었어요. 한 한달정도? 그래서 그때 되게 엄마가 그럴거면 도치를 왜 키우냐고 말했던 적이 있는데, 그때가 되게 후회스러워요. **그때 더 많이 잘해줄걸... 그냥 예쁜 거, 리본 하나라도 달아줘볼걸... 사진도 찍을 걸... 사진이 남아있는 게 또 없더라고요. 사진 되게 많이 찍었던 걸로 기억나는데 하나도 없고... 그리고 내가 되게 관심이 없었나?**이런 생각이 들어서 되게 아쉽고 미안했어요.

**Q. 사진같은 경우엔 드라이브와 같은 다른 곳에 미처 저장하지는 못한 건가요?**

A. 네. 그때는 이제 2G폰 쓰고 이제 막 스마트폰이 나올 때라 드라이브 같은 것도 몰랐고 찍어둔 것이 없어진 거에요. 그래서 되게 아쉽고... **그래서 지금 반려동물 강아지 사진은 많이 찍으려고 하고 있습니다.**

**Q. 반려동물과 함께 해서 행복했거나 기억에 남는 순간이 있으신가요?**

A. 이거는 모든 반려동물을 키우시는 분들이 공감하실 수 있는데, 그 아기를 멀리 이렇게 놓아둬요. 그리고 앞에 가족들이 다 있고 이리로 오라고 막 말을 해요. 근데 도치가 매 순간 저한테만 왔어요.

**Q. 오 도치가 말을 알아들었나요?**

A. 그니깐 말을 못 알아듣거든요? 걔는 분명히. 근데 제가 이렇게 오라고 하면 걔는 와요 저한테. 뭐 때문에 온지는 저도 몰라요. 아무튼 그 순간이 되게 기억에 남아요.

**Q. 그러면 반려동물과 이별하기 전으로 간다면 반려동물과 함께 하고 싶은 것은 무엇인가요?**

A. 전 그냥 얼굴이 보고싶어요. 왜냐하면 강아지처럼 산책을 갈 수 있는 것도 아니고 그러니깐... 그냥...

**Q. 그래도 지금은 어느정도 감정이 정리가 되신 것 같은데 도치랑 관련된 옛날 물건은 아직 가지고 계신가요?**

A. 도치랑 관련된 거는 이제 다 버린 것 같아요. 없는 것 같아요 집에는.

**Q. 버리게 되신 이유가 보면은 생각이 나서라든지 그런 이유에서인가요?**

A. **네. 그리고 저희도 집에 이제 그거를... 우리라고 하나? 그 집을, 플라스틱 통에서 키웠는데 엄청 한 2달 넘게? 한 3달 동안은 그 자리에 계속 놓아두었어요. 제가 계속 힘들어하고 엄마한테 치우지 말라고 하고 진짜 오래 그 자리에 있다가 이제 그래도 정리해야되지 않냐해서 그래서 치웠을 거에요.**   

**Q. 제가 듣기로는 반려동물과 이별 후, 반려동물과 비슷한 인형을 만들어서 가지고 계셨다고 들었는데 그 에피소드에 대해 설명해주실 수 있으실까요?**

A. 네. 제가 동물병원에 갔는데 제가 너무 어려서 그 때 충격을 되게 받은거에요. 동물이 막 경련을 일으키고 한 거에. 그래서 이제 아빠가 동물병원에 있고 엄마는 집에 동생과 저를 데리고 와서 있는데 아까 말한 케이지 안에 걔가 없다는 사실이 너무 허전하고 마음이 힘든 거에요. **그래서 그 때 왜 그랬는지 모르겠는데 재활용 통에 가서 요구르트 병을 최대한 비슷한 걸 막 찾은거에요 동생이랑 같이. 찾아서 막 이쑤시개 꼽아놓고 얘가 도치라고, 도치는 여기 다시 그대로 다시 올거라고.** 그랬었어요. 병원에 가 있었는데 어차피 치료받고 오면은 다시 올거니까... 도치오면은 바꿔주자 이렇게 그랬었어요.

**Q. 그럼 반려동물과 이별한 후에 어떻게 감정을 정리하셨는지 여쭤봐도 괜찮을까요?**

## 상태관리

- vue에서 pinia와 같이 상태관리를 하는 라이브러리가 리액트에도 존재

- state를 props로 자식에 전달해서 많이 사용하는데 여기저기서 사용해야 할 경우 접근하기 어려운 문제 발생 → 상태관리 라이브러리의 등장
- Redux
    - 많은 상태관리 라이브러리 중 가장 많이 사용하는건 Redux, 몇몇 회사에서는 리덕스 사용 경험이 있으면 우대하는 경우도 있음. (최근 올라왔던 채용 공고에도 redux, react query 사용한 프로젝트 3개 이상일 경우 우대, 기업 이름은 기억안남)
    - 리덕스는 알기로 리액트쪽 상태관리 라이브러리 1세대급이라 사용자가 많음
    - 하지만 사용 방법이 좀 어려워서 입문자들이 잘 사용하기 어려울 수 있음
    - 참고 영상 :
    
    [React 입문자들이 알아야할 Redux 쉽게설명 (8분컷)](https://www.youtube.com/watch?v=QZcYz2NrDIs)
    
    - 리덕스는 직접 설정해야 하는 부분이 많아 코드가 길어질 수 있음 → 설정 간소화한 리덕스 툴킷도 존재
- Recoil
    - 페이스북이 개발한 라이브러리
    - 리액트 훅과 비슷한 사용법 → 배우기 쉬운듯…? (리덕스보다 쉬워보여서 이거로 시작함)
    - atoms라는 상태 단위 사용, 상태 변경시 사용하는 컴포넌트들이 자동으로 리렌더링됨
    - 일반적으로 state/atoms.js에 작성
    
    ```jsx
    (index.js)
    //여기서 RecoilRoot 임포트하고 App을 감싸줘야 접근해서 사용 가능
    import { RecoilRoot } from "recoil";
    
    const root = ReactDOM.createRoot(document.getElementById("root"));
    root.render(
      <BrowserRouter>
        <RecoilRoot>
          <App />
        </RecoilRoot>
      </BrowserRouter>
    );
    serviceWorkerRegistration.register();
    reportWebVitals();
    ```
    
    ```jsx
    (atoms.js)
    //atom 만들고 형식 지정
    import { atom } from "recoil";
    import { recoilPersist } from "recoil-persist";
    
    const { persistAtom } = recoilPersist({
      key: "recoil-persist", // 이 키로 localStorage에 저장됩니다
      storage: localStorage, // localStorage를 사용하거나 sessionStorage로 변경 가능
    });
    
    export const userDataState = atom({
      key: "userDataState", // 고유한 키
      default: {
        nickName: "",
        exp: 0,
        walkedDay: 0,
        finishedCount: 0,
        pickedCount: 0,
        profile: "",
      },
      effects_UNSTABLE: [persistAtom],
    });
    
    export const userIdState = atom({
      key: "userIdState",
      default: {
        id: 0,
      },
      effects_UNSTABLE: [persistAtom],
    });
    
    export const geolocationState = atom({
      key: "geolocationState",
      default: {
        longitude: 0,
        latitude: 0,
      },
      effects_UNSTABLE: [persistAtom],
    });
    
    export const locationState = atom({
      key: "locationState",
      default: "",
      effects_UNSTABLE: [persistAtom],
    });
    ```
    
    ```jsx
    (MyPage.jsx)
    //만든 atom 사용 예시
    
    import { useRecoilState, useRecoilValue, useResetRecoilState } from "recoil";
    //useRecoilState는 useState훅처럼 상태 변수를 만들고, 변수 값 설정하는 함수 생성, ()내에 들어가는 atom의 값을 저장
    //useRecoilValue는 ()내 atom의 값을 가져와 사용
    //useResetRecoilState는 atom의 상태를 초기화
    
    ...
    const resetUserData = useResetRecoilState(userDataState);
    const userId = useRecoilValue(userIdState);
    const [userData, setUserData] = useRecoilState(userDataState);
    
    //사용자 정보 atoms에 저장. useState와 사용방법이 유사하죠?
    //사용자 정보를 userId로 요청하는 함수로 받아와fetchedData에 저장, 이후 setUserData로 userDataState atom에 저장해 전역적으로 사용
    useEffect(() => {
      const fetchData = async () => {
        const fetchedData = await fetchUserData(userId); 
        console.log(fetchedData);
        setUserData(fetchedData);
      };
    
      fetchData();
    }, []);
    
    //카카오 로그아웃시 저장된 사용자 정보 삭제
    const handleLogout = () => {
      if (window.Kakao.Auth) {
        window.Kakao.Auth.logout(() => {
          console.log("kakao 로그아웃 성공");
          localStorage.removeItem("token");
          resetUserData(); // 저장된 atom 비워줌
          console.log("로컬스토리지 토큰 삭제 및 상태 초기화");
          navigate("/");
        });
      }
    };
    ```
    
    - 단점 : 이유는 모르겠는데 새로고침 누르면 저장된 atoms의 값들이 다 날아가는 문제 발생
        - 그래서 찾아보니 recoil-persist사용해서 로컬스토리지나 세션스토리지에 저장해서 사용한다고함
        - 위에 있는 atoms.js도 recoilPersist를 사용한 코드(로컬스토리지에 저장됨)
        - 그러면 로컬스토리지를 그냥 사용하지 왜 리코일을 쓰는가?
            
            GPT답변 : Recoil을 사용하는 이유는 Recoil이 제공하는 상태 관리의 강력한 기능과 React와의 통합이 매우 뛰어나기 때문입니다. 로컬 스토리지에 상태를 저장한다고 해서 Recoil의 필요성이 사라지는 것은 아닙니다. 오히려 Recoil의 기능을 확장하여 더 나은 사용자 경험을 제공할 수 있습니다. Recoil과 로컬 스토리지를 결합하면 중앙 집중식 상태 관리와 상태 영속성의 이점을 동시에 누릴 수 있습니다.

- 데이터 페칭 및 상태 관리 라이브러리
    - 서버 상태를 관리하고 데이터를 동기화하기 위한 라이브러리라는데… 써봐야 제대로 이해할듯
    - 프론트엔드 직군에서 리액트 쿼리 사용 경험을 우대 사항으로 둔 기업들 간간히 존재
    
    > (kakao)2021 - 카카오페이 프론트엔드 개발자들이 React Query를 선택한 이유」 세줄요약
    > 
    1. React Query는 React Application에서 서버 상태를 불러오고, 캐싱하며, 지속적으로 동기화하고 업데이트하는 작업을 도와주는 라이브러리입니다.
    2. 복잡하고 장황한 코드가 필요한 다른 데이터 불러오기 방식과 달리 React Component 내부에서 간단하고 직관적으로 API를 사용할 수 있습니다.
    3. 더 나아가 React Query에서 제공하는 캐싱, Window Focus Refetching 등 다양한 기능을 활용하여 API 요청과 관련된 번잡한 작업 없이 "핵심 로직"에 집중할 수 있습니다.
    
    > react-query 장점
    > 
    - 캐싱
    - get을 한 데이터에 대해 update를 하면 자동으로 get을 다시 수행한다. (예를 들면 게시판의 글을 가져왔을 때 게시판의 글을 생성하면 게시판 글을 get하는 api를 자동으로 실행 )데이터가 오래 되었다고 판단되면 다시 get (invalidateQueries)
    - 동일 데이터 여러번 요청하면 한번만 요청한다. (옵션에 따라 중복 호출 허용 시간 조절 가능)
    - 무한 스크롤 (Infinite Queries (opens new window))비동기 과정을 선언적으로 관리할 수 있다.
    - react hook과 사용하는 구조가 비슷하다.

    ### QueryClientProvier, QueryClient
    
    - 리액트 쿼리 사용시 `QueryClientProvider`을 최상단에서 감싸줘야 한다
    - 쿼리 인스턴스를 생성 후 `client={queryClient}`를 작성
        - `QueryClient` 인스턴스를 리액트 컴포넌트 트리에 제공, 하위 컴포넌트들이 쿼리 클라이언트를 통해 데이터 페칭 및 상태 관리
    - `QueryClient`는 상태 관리, 캐싱 메커니즘 제공하는 객체
        - 쿼리 캐싱
        - 데이터 동기화 및 갱신(캐시된 데이터를 자동으로 갱신,
        - 쿼리 상태 관리(로딩중, 성공, 오류)
        - 전역 쿼리 설정 (특정 쿼리 무효화, 재실행 기능 제공 → 데이터 변경시 특정 쿼리 수동으로 갱신해야 하는 상황에서 유용)
    
    ```jsx
    import { QueryClient, QueryClientProvider } from "react-query";
    
    const queryClient = new QueryClient();
    
    export default function App() {
      return (
        <QueryClientProvider client={queryClient}>
          <Home />
        </QueryClientProvider>
      );
    }
    ```
    
    ### useQuery
    
    - `useQuery`는 서버로부터 데이터 가져와 데이터를 캐시에 저장, 데이터의 상태를 관리하는 역할
    - `useQuery` 를 사용해 컴포넌트 내에서 간편하게 데이터 페칭, 상태 관리 가능
    - 주요 파라미터
        - `queryKey` 쿼리 고유 식별자, 문자열 or 배열. 이 키를 통해 캐시에서 데이터 식별, 관리
        - `fetchFunction` 데이터를 가져오는 비동기 함수
    - 반환 값
        - isLoading
        - error
        - data
        - isFetching :
        - status : 쿼리의 현재 상태를 문자열로 나타냄 (’loading’, ‘error’, ‘success’)
    - 옵션(세 번쨰 인자)
        - enabled : 쿼리 실행 여부 결정, false 설정시 쿼리 자동으로 실행되지 않음
            - id가 존재할 때만 쿼리 요청
            
            ```tsx
            const { data } = useQuery(
              ['todos', id],
              () => fetchTodoById(id),
              {
                enabled: !!id,
              }
            );
            ```
            
        - staleTime : 데이터가 신선한 상태로 유지되는 시간
            - fresh 상태에서는 쿼리가 다시 mount 되어도 fetch가 실행되지 않는다.
        - cacheTime : 캐시된 데이터가 메모리에 유지되는 시간
        - refetchOnWIndowFocus : 창이 다시 포커스 받을 떄 쿼리 재실행 여부. 기본값은 true
    
    ```jsx
    import { useQuery } from 'react-query';
    import axios from 'axios';
    
    const fetchUser = async () => {
      const { data } = await axios.get('/api/user');
      return data;
    };
    
    function UserComponent() {
      const { data, error, isLoading } = useQuery('user', fetchUser);
    
      if (isLoading) return <span>Loading...</span>;
      if (error) return <span>Error: {error.message}</span>;
    
      return (
        <div>
          <h1>{data.name}</h1>
          <p>{data.email}</p>
        </div>
      );
    }
    ```
    
    ### useMutation
    
    - 서버에 데이터를 생성, 업데이트, 삭제시 사용 (CUD)
    - 쿼리와 달리 뮤테이션은 사용자가 명시적으로 실행
    - 비동기 작업을 처리하고, 상태를 관리하는데 도움
    
    ```jsx
    const { data, isLoading, mutate, mutateAsync } = useMutation(mutationFn, options);
    ```
    
    1. mutation 함수 
        1. 데이터를 변경하는 비동기 함수(api요청 함수)
    2. option 객체
        1. `onMutate` : mutationFn가 실행되기 전에 실행되고 mutation함수가 받을 동일한 변수가 전달, `optimistic update` 사용시 유용
        2. `onSuccess` : mutation이 성공하고 결과 전달시 실행
        3. `onError` : mutation이 error 만났을 때 실행
        4. `onSettled` : 성공이든 실패든 전달
    3. 리턴값
        - **mutate**: 뮤테이션을 트리거하는 함수입니다.
        - **mutateAsync**: 뮤테이션을 트리거하고 프로미스를 반환하는 함수입니다.
        - **isLoading**: 뮤테이션이 진행 중인지 여부를 나타냅니다.
        - **isSuccess**: 뮤테이션이 성공적으로 완료되었는지 여부를 나타냅니다.
        - **isError**: 뮤테이션이 실패했는지 여부를 나타냅니다.
        - **error**: 뮤테이션 중 발생한 오류 객체입니다.
        - **data**: 성공적으로 완료된 후 반환된 데이터입니다.
    
    ```jsx
    import { useMutation, useQueryClient } from 'react-query';
    import axios from 'axios';
    
    const addUser = async (newUser) => {
      const { data } = await axios.post('/api/users', newUser);
      return data;
    };
    
    function AddUserComponent() {
      const queryClient = useQueryClient();
    
      const mutation = useMutation(addUser, {
        onSuccess: () => {
          // 성공적으로 추가된 후 'users' 쿼리를 무효화하여 데이터를 갱신합니다.
          queryClient.invalidateQueries('users');
        },
        onError: (error) => {
          console.error('Error adding user:', error);
        },
        onSettled: () => {
          console.log('Mutation has settled');
        },
        retry: 2, // 실패시 2번 재시도
        retryDelay: 1000, // 재시도 사이의 지연 시간을 1초로 설정
      });
    
      const handleAddUser = () => {
        mutation.mutate({ name: 'New User' });
      };
    
      return (
        <div>
          <button onClick={handleAddUser}>Add User</button>
          {mutation.isLoading && <p>Adding user...</p>}
          {mutation.isError && <p>Error adding user</p>}
          {mutation.isSuccess && <p>User added successfully!</p>}
        </div>
      );
    }
    ```
    
    ```jsx
    const { mutate, isLoading, isError, error, isSuccess } = useMutation(newTodo => {
      return axios.post<TodoType>('/todos', newTodo);
    });
    ```
    
    ### invalidateQueries
    
    - 특정 쿼리 무효화하고 캐시된 데이터를 갱신하는 역할
    - 데이터가 변경된 후 쿼리를 다시 실행하도록 트리거, 최신 데이터 가져옴
    - useMutation과 사용
    - 쿼리 무효화 이유
        - 데이터 일관성 유지
        - 자동 갱신
        - 캐시 무효화 → 메모리 측면 이점
    
    ```jsx
    //사용자 추가 후 유저 목록 갱신
    import { useMutation, useQueryClient } from 'react-query';
    import axios from 'axios';
    
    const addUser = async (newUser) => {
      const { data } = await axios.post('/api/users', newUser);
      return data;
    };
    
    function AddUserComponent() {
      const queryClient = useQueryClient();
    
      const mutation = useMutation(addUser, {
        onSuccess: () => {
          // 'users' 쿼리를 무효화하여 데이터를 최신 상태로 갱신합니다.
          queryClient.invalidateQueries('users');
        },
      });
    
      const handleAddUser = () => {
        mutation.mutate({ name: 'New User' });
      };
    
      return (
        <div>
          <button onClick={handleAddUser}>Add User</button>
          {mutation.isLoading && <p>Adding user...</p>}
          {mutation.isError && <p>Error adding user</p>}
          {mutation.isSuccess && <p>User added successfully!</p>}
        </div>
      );
    }
    ```

    ## 🌟 optimistic updates !
    
    > 💡 mutation 의 optimistic update 는 사용자가 어떠한 액션을 발생시켰을 때 요청이 성공했는지 실패했는지 아직 알 수 없는 상태에서 성공할 것이라고 낙관적으로 가정하고 사용자가 보고 있는 UI 를 먼저 변화시켜주는 것을 말한다. 사용자에게 빠른 경험을 제공할 수 있다는 큰 장점이 있다!
    > 
    
    ```tsx
    import { AxiosError } from 'axios';
    import { useMutation, UseMutationResult, useQueryClient } from 'react-query';
    import { addTodo } from 'src/api/todos';
    import { TodoType } from 'src/types/todoType';
    import { queryKeys } from 'src/types/commonType';  // useQuery 실용 편 참조
    
    export default function useAddTodoMutation(): UseMutationResult<
      TodoType,
      AxiosError,
      TodoType,
      {
        previousTodos: TodoType[] | undefined;
      }
    > {
      const queryClient = useQueryClient();
      return useMutation(addTodo, {
        onMutate: async (newTodo: TodoType) => { // mutate가 호출될 때
          // 쿼리를 확실하게 취소하고
          await queryClient.cancelQueries(queryKeys.todos);
          // 쿼리 상태를 가져온다(이전 값 스냅샷)
          const previousTodos = queryClient.getQueryData<TodoType[]>(queryKeys.todos);
    
          if (previousTodos) {
            // previousTodos 가 있으면 setQueryData 를 이용하여 즉시 새 데이터로 업데이트 해준다.
            queryClient.setQueryData<TodoType[]>(queryKeys.todos, (old) => [
              ...(old as TodoType[]),
              newTodo,
            ]);
          }
          return { previousTodos }; // 이전 값을 리턴한다
        },
        onError: (
          err: AxiosError,
          variables: TodoType,
          context?: { previousTodos: TodoType[] | undefined }) => {
          if (context?.previousTodos) { // error 를 만났을 경우 onMutate에서 반환된 값으로 다시 롤백시켜준다.
            queryClient.setQueryData<TodoType[]>(queryKeys.todos, context.previousTodos);
          }
        },
        onSettled: () => { // mutation이 끝나면 (성공유무 상관없이) 쿼리를 무효화 처리하고 새로 가져온다.
          queryClient.invalidateQueries(queryKeys.todos);
        },
      });
    }
    ```
    
    - `queryClient.cancelQuerie` : 발신 쿼리를 취소하는 데 사용할 수 있다. (optimistic update를 덮어쓰지 않도록)
    - `queryClient.getQueryData` : 기존 쿼리의 상태를 가져오는 동기 함수. 쿼리가 존재하지 않으면 `undefined` 를 반환.
    - `queryClient.setQueryData` : 쿼리의 캐시된 데이터를 즉시 업데이트할 수 있는 동기 함수. 쿼리가 존재하지 않으면 생성된다.
    
    optimistic updates 를 이용하면 사용자가 api 요청 성공 유무를 기다리지 않고 바로 UI 단에서 변화를 확인할 수 있어 유용하다.
    
    ex) 페이스북 좋아요를 누른다고 가정할 때 좋아요를 누르거나 취소할 때마다 api 결과 기다려야 한다면 매우 답답할 것이다. 사용자가 좋아요를 마구 눌렀다 취소할 수 도 있는데 이럴 때 특히 유용하다.

## 1. 디렉토리 구조

```
└─ src
 ├─ components
 ├─ assets
 ├─ hooks (= hoc)
 ├─ pages
 ├─ constants
 ├─ config
 ├─ styles
 ├─ services (= api)
 ├─ utils
 ├─ contexts
 ├─ App.js
 └─ index.js
```

- **components**
    
    재사용 가능한 컴포넌트들이 위치하는 폴더입니다.
    
    컴포넌트는 매우 많아질 수 있기 때문에 이 폴더 내부에서 하위폴더로 추가로 분류하는 경우가 많습니다.
    
- **assets**
    
    이미지 혹은 폰트와 같은 파일들이 저장되는 폴더입니다.
    
    이미지와 같은 파일들을 public에 직접 넣는 경우도 있는데 **둘의 차이는 컴파일시에 필요한지 여부**입니다.
    
    파비콘과 같이 `index.html`내부에서 직접 사용하여 컴파일 단계에서 필요하지 않은 파일들은 public에
    
    반면, 컴포넌트 내부에서 사용하는 이미지 파일인 경우 이 assets 폴더에 위치시켜야 합니다.
    
- **hooks (= hoc)**
    
    커스텀 훅이 위치하는 폴더입니다.
    
- **pages**
    
    react router등을 이용하여 라우팅을 적용할 때 페이지 컴포넌트를 이 폴더에 위치시킵니다.
    
- **constants**
    
    공통적으로 사용되는 상수들을 정의한 파일들이 위치하는 폴더입니다.
    
- **config**
    
    config 파일이 많지 않은 경우 보통 최상위에 위치시켜놓지만 여러개의 config 파일이 있을 경우 폴더로 분리하기도 합니다.
    
- **styles**
    
    css 파일들이 포함되는 폴더입니다.
    
- **services (= api)**

보통 api관련 로직의 모듈 파일이 위치하며 auth와 같이 인증과 관련된 파일이 포함되기도 합니다.

- **utils**
    
    정규표현식 패턴이나 공통함수 등 공통으로 사용하는 유틸 파일들이 위치하는 폴더입니다.
    
- **contexts(store)**
    
    contextAPI를 사용할 때 관련 파일들이 위치하는곳으로 상태관리를 위해 contextAPI 대신 redux를 사용 할 경우 폴더 이름을 `store`로 사용하기도 합니다.

export default ArrowIcon = () ⇒ {

# 오픈비두 화면공유 코드 (에러있는데 아까워서 보관용)

```jsx
// 화면 공유 (하나의 파일에 다씀 OpenViduApp.tsx) 

  const [screensharingStream, setScreensharingStream] = useState<StreamManager | undefined>(
    undefined
  );

  const [screensharing, setScreensharing] = useState<boolean>(false);

  const publishScreenShare = () => {
    if (!screensharing && session) {
      // 화면 공유를 시작할 때
      const OVScreen = new OpenVidu();
      const sessionScreen = OVScreen.initSession();

      createToken(mySessionId)
        .then((token) => {
          sessionScreen
            .connect(token)
            .then(() => {
              const publisherScreen = OVScreen.initPublisher(undefined, {
                videoSource: 'screen', // 화면 공유
                publishAudio: false, // 오디오 없음
                mirror: false,
              });

              publisherScreen.once('accessAllowed', () => {
                console.log('화면 공유 접근 허용됨');
                setScreensharingStream(publisherScreen);
                setScreensharing(true);

                // 화면 공유 스트림을 구독
                const screenSubscriber = sessionScreen.subscribe(
                  publisherScreen.stream,
                  'subscriber'
                );
                setSubscribers((prevSubscribers) => [...prevSubscribers, screenSubscriber]);

                // 화면 공유 스트림이 종료되었을 때 처리
                publisherScreen.stream
                  .getMediaStream()
                  .getVideoTracks()[0]
                  .addEventListener('ended', () => {
                    console.log('화면 공유 종료됨');
                    sessionScreen.unpublish(publisherScreen);
                    setSubscribers((prevSubscribers) =>
                      prevSubscribers.filter((sub) => sub !== screenSubscriber)
                    ); // 구독자 배열에서 제거
                    setScreensharingStream(undefined);
                    setScreensharing(false);
                  });

                // 화면 공유 스트림을 세션에 게시
                sessionScreen.publish(publisherScreen);
              });

              publisherScreen.once('accessDenied', () => {
                console.warn('화면 공유: 접근 거부됨');
              });
            })
            .catch((error) => {
              console.error('화면 공유 연결 오류:', error);
            });
        })
        .catch((error) => {
          console.error('화면 공유 토큰 생성 오류:', error);
        });
    } else {
      // 화면 공유를 중지할 때
      if (screensharingStream && session) {
        session.unpublish(screensharingStream as Publisher);
        setSubscribers((prevSubscribers) =>
          prevSubscribers.filter((sub) => sub !== screensharingStream)
        ); // 구독자 배열에서 제거
        setScreensharingStream(undefined);
      }
      setScreensharing(false);
    }
  };

       {screensharingStream !== undefined ? (
                <div className='box-border stream-container col-md-3'>
                  <UserVideoComponent streamManager={screensharingStream} />
                </div>
              ) : null}

       <CircleButton
                theme={screensharing ? 'white' : 'hover'}
                onClick={publishScreenShare}
                icon={'share'}
                disabled={screensharing}
              />
```
